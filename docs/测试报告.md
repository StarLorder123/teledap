## 2. Bug 发现与修复

在测试过程中发现并修复了以下问题：

### Bug 1: `customLaunchSetupCommands` 字段不存在（GDB 远程调试完全失效）

**根因**: teledap 向 codelldb 发送 `customLaunchSetupCommands` 字段，但 codelldb 不识别此字段（serde 静默忽略未知字段）。

**影响**: 在 CLI 和 MCP 模式下，GDB 远程调试的 `gdb-remote` 命令从未到达 LLDB，导致远程连接失败。同时命令格式为 `[{"text": "..."}]` 也不正确，codelldb 期望 `["..."]`（`Vec<String>`）。

**修复**: 将 `customLaunchSetupCommands` 改为 codelldb 实际支持的 `processCreateCommands`，命令格式从对象数组改为字符串数组。

**修改文件**:
- `src/cli.rs:169` — CLI 模式
- `crates/debug-bridge/src/handlers/lifecycle.rs:127` — MCP 模式

**修改前**（错误）:
```rust
launch_extra["customLaunchSetupCommands"] =
    serde_json::json!([{"text": format!("gdb-remote {remote}")}]);
```

**修改后**（正确）:
```rust
launch_extra["processCreateCommands"] = serde_json::json!([format!("gdb-remote {remote}")]);
```

**参考**: codelldb 源码 `E:\Code\codelldb\src\codelldb-types\src\lib.rs:224-253` 中 `LaunchRequestArguments` 结构体确认了正确的字段名；`debuggee\.vscode\launch.json:153-157` 和 `MANUAL.md:409-410` 提供了正确的使用示例。

### Bug 2: `adapterID` 字段名大小写错误（initialize 握手失败）

**根因**: DAP 协议规范要求 `adapterID`（大写 ID），但 serde 的 `#[serde(rename_all = "camelCase")]` 将 `adapter_id` 转换为 `adapterId`（小写 id）。`adapterID` 是 initialize 请求的**必填字段**，codelldb 收到 `adapterId` 后因找不到必填字段而返回 `success: false, message: "Malformed message"`。

**影响**: 所有 initialize 握手均失败，调试会话无法建立。CLI 和 MCP 模式均受影响。

**修复**: 在 `adapter_id` 和 `client_id` 字段上添加显式的 `#[serde(rename)]` 覆盖。

**修改文件**: `crates/dap-types/src/requests.rs:55,61`

**修改前**:
```rust
#[serde(default, skip_serializing_if = "Option::is_none")]
pub client_id: Option<String>,
// ...
#[serde(default, skip_serializing_if = "Option::is_none")]
pub adapter_id: Option<String>,
```

**修改后**:
```rust
#[serde(default, skip_serializing_if = "Option::is_none", rename = "clientID")]
pub client_id: Option<String>,
// ...
#[serde(default, skip_serializing_if = "Option::is_none", rename = "adapterID")]
pub adapter_id: Option<String>,
```

**参考**: codelldb 的单元测试（`E:\Code\codelldb\src\adapter-protocol\src\lib.rs:224`）使用 `clientID`、`adapterID` 等字段名。DAP 协议 JSON Schema（`debugAdapterProtocol.json:906`）也明确标记 `"adapterID"` 为 required 字段。

### Bug 3: CLI 事件流程死锁（等待 `initialized` 事件在 launch 之前）

**根因**: CLI 代码在发送 `launch` 请求之前等待 `initialized` 事件，但 codelldb 的 `initialized` 事件是在 `launch` 请求处理期间（`handle_launch()` 内部）才发送的。形成死锁：CLI 等事件 → 不发 launch → codelldb 不发事件。

**影响**: CLI 模式卡在 "Waiting for 'initialized' event..." 后无响应。

**修复**: 调整事件流程——先用 fire-and-forget 方式（`send_request_nb`）发送 launch，再等待 `initialized` 事件。

**修改文件**: `src/cli.rs:127-190`（整个步骤 3–4 重新编排）

**正确流程**:
```
1. send initialize → await response
2. send launch (fire-and-forget) ← codelldb processes, sends initialized event
3. wait for initialized event ← arrives during launch processing
4. set breakpoints
5. send configurationDone → codelldb completes launch (actually starts process)
6. event loop → handle stopped/output/exited/terminated
```

### Bug 4: `continued` 事件在 Running 状态触发错误

**根因**: `configurationDone` 后状态变为 `Running`，随后 `continued` 事件尝试再次转换到 `Running`，但状态机不允许 `Running → Running`（非法自循环）。

**影响**: 每次 `continued` 事件都产生错误日志。

**修复**: `continued` 事件处理中增加状态判断——如果已经是 `Running`，跳过转换。

**修改文件**: `crates/debug-session/src/session.rs:569-586`

### 改进: 新增 `--cli` 标志

**根因**: PowerShell 中 `std::io::stdin().is_terminal()` 返回 `false`（stdin 为管道），导致程序误判为 MCP 模式而阻塞等待 JSON-RPC 输入。

**修改文件**: `src/main.rs:19-20`, `src/cli.rs:37-39`

**使用方式**:
```bash
teledap --cli --codelldb-path ... --elf-path ...
```

```

    │  start()
    ▼
Connected
    │  initialize()
    ▼

    │  wait for initialized event
    │  setBreakpoints()
    │  configurationDone()
    ▼
Running                     ← 调试目标进程启动
    │  continued event      ← 进程开始执行
    │  output events        ← printf 输出到达
    │  exited event         ← 进程正常退出 (code 0)
    ▼
Disconnected
    │  shutdown()
    ▼
[Session end]
```

### 4.3 注意事项

- **`Received response with no pending request` warning**: launch 使用了 send_request_nb（fire-and-forget），codelldb 的 launch 响应到达时无匹配的 pending request。这是预期行为，不影响功能。
- **断点 position**: `BP id=Some(0), verified=false, line=None` — 使用 `line: 0` 设置断点（非真实行号），断点未验证通过。使用真实行号可解决。
- **`--cli` 标志必需**: PowerShell 等非终端 stdin 环境需要 `--cli` 强制 CLI 模式。
